Elastic Best-Practice Reference
Shard Size
Target 10–50 GB per shard. The sweet spot is 20–40 GB. Shards below 10 GB create excess JVM overhead; above 50 GB cause slow recoveries and merges.
Hot / Warm Density
Hard limit: 1,000 shards per node on hot and warm tiers (Elasticsearch 8.3+ default for cluster.max_shards_per_node). The older 20 shards/GB-heap rule is deprecated.
Cold / Frozen Density
Cold: 1,000 shards/node. Frozen: 3,000 shards/node. Cold/frozen nodes carry no indexing pressure — only search metadata — so much higher densities are safe.
Cluster Shard Limit
Elastic guideline: 1,000 shards × number of non-frozen data nodes. This scales with your cluster size. Exceeding the limit increases cluster state overhead and can degrade master stability.
Cluster Heap Rule
Across hot + warm data nodes, maintain no more than 20 shards per heap-GB of total data-node heap. This is the cluster-wide equivalent of the per-node ceiling.
ILM Rollover
Use ILM rollover with a max_primary_shard_size of 50 GB or max_age of 1 day. This bounds shard size automatically without over-splitting at ingest time.
